CX Copilot
Slack 기반 CS 자동응답 챗봇

안녕하세요, 개발팀 여러분!

이 문서는 CX 자동화를 위한 CX Copilot 챗봇 제작 프로젝트의 기획서입니다. 읽어보신 후, 함께 진행할 수 있는 부분은 함께, 혹은 개인이 해야 하는 프로젝트라면 혼자서라도 진행해 주셨으면 합니다.

이 프로젝트는 생각보다 급한 상황이라 판단되어 급하게 요청드리게 되었습니다. 다음 주부터는 프로덕트 개발이 주요 우선순위가 되면서 팀 여유가 줄어들 것 같고, 그렇다고 이걸 미루자니 매일 채널톡에 쌓이는 CS 문의에 아무런 개선이 이뤄지지 않는 상황이 계속되고 있습니다.

이 챗봇이 해결하려는 건 하나입니다. 반복되는 문의를 자동으로 처리해서, CX 매니저가 정말 어렵고 중요한 케이스에만 집중할 수 있게 하는 것.

읽어보시고, 혹시 더 자세한 기획이나 논의가 필요한 건 함께 이야기해 주시면 감사하겠습니다!

01

핵심 요약

채널톡으로 누적된 512건의 CS 문의 중 60% 이상이 8가지 반복 패턴으로 이루어져 있습니다. CX 매니저 1명이 전체의 81%를 단독으로 처리하고 있으며, 마이그레이션 시점에는 처리되지 못한 문의가 60건까지 쌓였습니다.

저희가 만들 것은 사내 슬랙에서 @cx-copilot을 부르거나 /cs를 입력해 아래 두 가지를 처리하는 AI 챗봇입니다.

  • 정책 질문 즉시 답변 — "예치금 입금 조건이 뭐야?" 같은 정책·규칙·매뉴얼 질문에 출처와 함께 바로 답변
  • 시스템 작업 자동 처리 — 비밀번호 초기화, 잔액 이전 등 실제 시스템 변경을 슬랙 버튼 한 번으로

02

배경 — 왜 지금인가

누적 데이터

총 512건의 CS 문의를 분석한 결과, 마지막 분석 시점 단일 기간 기준 103건이 역대 최고치였고, 그중 60건이 진행 중 적체 상태였습니다. 마이그레이션이라는 일회성 이벤트로 폭증했지만, 동시에 CX 운영의 구조적 한계가 드러난 사건이기도 합니다.

본질적 문제 3가지

영역문제구체 사례
반복성동일 패턴이 계속 반복비밀번호 분실, QR 사용 불가, 전자계약서 미수신
단일 의존성매니저 1명이 81.3% 단독 처리 → 단일 장애점매니저 휴가 시 적체, 노하우가 채널톡에만 쌓임
엣지 케이스시스템 한계로 개발팀 호출 발생쌍둥이 인증, 가상번호 인증, 사업자명 특수문자
추가로 확인된 패턴
반복 문의 중 상당수는 "정책이나 규칙을 몰라서"입니다. 예치금 충전 한도, 환불 정책, 최소 주문 수량 같은 질문은 SOP에 답이 명확히 있음에도 매번 매니저에게 문의합니다. 이 점이 지식 Q&A 모드를 만들게 된 직접적인 이유입니다.

왜 지금인가

세 가지 요인이 동시에 맞물렸습니다.

  1. 운영 한계 노출 — 마이그레이션 적체로 처리 병목이 가시화됨
  2. 모델 품질 임계 도달 — Claude Sonnet 4.6 / Haiku 4.5가 한국어 도메인 대화에서 충분한 수준에 도달
  3. 워크플로 정착 — 사내 슬랙이 CX팀의 일상 도구로 자리잡음

반복 응답을 자동화하고, 어려운 케이스에만 집중하고 싶다

입사 초기, 정책 질문마다 매니저에게 확인 중

👨‍🏫 원장님 — 민수
채널톡으로 문의, 빠른 답변을 원함

📊 PM — 지원
CS 핫스팟을 파악해야 함
정량 데이터로 우선순위를 판단하고 싶다

유저 스토리

US-1. 신입 CX가 정책을 매니저 없이 알고 싶다

  • @cx-copilot 예치금 입금 조건 한 줄로 답변 + 노션 SOP 링크 + 마지막 수정일 제공
  • 답변에 인용 출처 및 신뢰도 표시 (예: "🟢 SOP에 명확히 정의됨")
  • 출처가 오래됐으면 솔직히 표시 ("⚠️ 출처가 오래되었을 수 있음")

US-2. 매니저가 반복 문의를 한 번에 끝내고 싶다

  • @cx-copilot 비번 초기화 수학의신학원 한 줄로 학원 식별 → 비번 발급 → 알림톡 발송 완료
  • 챗봇 응답 30초 이내, 실행 버튼 1번으로 처리

US-3. 개발자가 일회성 DB 수정 요청을 받지 않는다

  • 5개 엣지 케이스가 슬랙에서 자가 처리 가능
  • 모든 우회 액션에 감사 로그 + 즉시 롤백 지원

US-4. 매니저가 챗봇이 자주 틀리는 영역을 알 수 있다

  • 정기 리포트에 질문 유형별 👎 비율 자동 집계
  • 👎 5건 누적 시 슬랙 알림 발송

US-5. PM이 답변 스타일을 데이터로 비교할 수 있다

  • 프롬프트 변경 → 트래픽 분할 → 통계 유의성 → 의사결정까지 운영 도구에서 가능

US-6. 보안 담당자가 모든 민감 액션을 추적할 수 있다

  • Mutating 액션 영구 보존, 즉시 조회 가능

챗봇이 처리하는 두 가지 질의 유형

이 챗봇은 본질적으로 두 가지 종류의 일을 합니다. 처리 방식도, 측정 지표도 다릅니다.

항목🔍 유형 1: 정책 질문 답변⚡ 유형 2: 시스템 작업
하는 일회사 규칙·정책·매뉴얼에 대한 질문에 답변시스템 상태 변경 (DB 수정, 외부 처리)

답변 품질 규칙

정책 질문 답변은 아래 6가지 원칙을 반드시 따릅니다. 매니저 호출이 필요한 경우 7번(운영 모드) 흐름을 따릅니다.

  1. 출처 명시 필수 — 모든 답변에 SOP 섹션 명시
  2. 신뢰도 표시 — 🟢 명확히 정의됨 / 🟡 일부 추정 포함 / 🔴 출처 부재
  3. 모르면 모른다고 — 출처가 없거나 모호하면 "확인 필요" + 매니저 호출 버튼 (→ 매니저 검토 모드)
  4. 신선도 경고 — 출처가 오래됐으면 "⚠️ 출처가 오래되었을 수 있음" 자동 표시
  5. 충돌 시 보고 — 두 출처가 충돌하면 양쪽 모두 보여주고 매니저에게 SOP 정리 요청
  6. 검증 가능성 — 모든 답변에 [📚 원문 보기] 버튼으로 출처 원문 즉시 열람 가능

A/B 테스트 전략

AI 답변 품질은 직관으로 결정할 수 없습니다. 모든 의사결정을 데이터로 합니다. (Intercom Fin, ClearFeed 등 해외 AI 고객지원 서비스도 같은 방식으로 운영합니다.)

실험 백로그

실험가설핵심 지표의사결정
1. 인용 형식
Inline vs Footer		 Inline 인용이 클릭률을 높이지만 가독성을 해친다 출처 클릭률 Inline이 클릭률 30%↑이면 채택
2. 답변 길이
요약형 vs 자세히		 요약형이 빠른 처리를 가능하게 하지만 후속 질문이 늘어난다 후속 질문 비율 "요약형 + [자세히] 버튼" 하이브리드 채택 여부
3. 답변 톤
공감형 vs 간결형		 공감형이 만족도는 높지만 처리 시간이 길어진다 문의 처리 소요 시간 톤이 처리 시간에 미치는 영향 측정 후 표준 확립
4. 모델
Sonnet 4.6 vs Haiku 4.5		 단순 FAQ는 Haiku가 Sonnet 대비 품질 -2%p 이내, 비용 80% 절감 자동 해결률 확신도 기반 자동/수동 분기 정책 수립
5. 자동화 수준
수동 확인 vs 자동 실행		 Confidence > 0.95이면 사람 확인 없이 즉시 실행해도 안전 잘못된 액션 발생률 안전 기준: 0건 (발생 즉시 중단)
6. 문서 검색 범위
top-3 vs top-8 비교		 top-8이 정확도 향상 + 응답 시간 큰 차이 없음 매니저 평가 정확도 응답 시간 +500ms 이내 조건부 채택
7. UX 형식
카드 vs 일반 텍스트		 카드 UI가 시각적으로 좋지만 클릭률은 낮음 액션 버튼 클릭률 초기 데이터 수집 후 결정
8. 후속 질문 자동 제안 관련 질문 자동 제안이 추가 학습을 유도한다 챗봇 사용 빈도 초기 데이터 수집 후 결정
