반복 패턴 — 무엇이 되풀이되는가

각 패턴을 클릭하면 접고 펼 수 있습니다.

1
온보딩 계정/기관 생성이 사실상 매일
22건+
  • DB 분석에선 안 잡혔던 최대 덩어리. 학원명 / 사업자번호 / 원장명 / 전화 4개만 주면 끝나는 완전 정형 요청
  • "퇴근하셨는데 죄송… 계정 생성 부탁드려요"가 거의 매일 반복
개선: 셀프 온보딩 또는 챗봇 트리거 잡 → 단일 항목으로 소통량 최다 제거 (잔액이전 취소보다 상위)
2
"안 보여요/조회 안돼요"의 원인은 거의 정해져 있다
10건+
  • 클레모에서 배포한 카드 → 티처스에 안 보임 (데이터 출처 이원화)
  • 원장이 본인이 직접 만든 중복 학원에 들어가서 조회 (예: 잉글리쉬타임어학원 vs 잉글리쉬타임영어학원)
개선: 챗봇이 카드번호/학원만 받아 "이 카드는 클레모 배포건이라 티처스 미표시" 같은 결정적 답 즉답 → 진선·예원 모두 우회
3
에러가 소수 유형으로 수렴한다
22건 / 원인 5종
  • "유효하지 않은 기관입니다" → 학원명에 (주)/㈜ 특수문자 또는 쉼표(,) 포함
  • 로그인/비번 실패 → 카카오 소셜 가입인데 일반 로그인 시도
  • 수동매핑 "서버 내부 오류" → 매핑 로직 버그
  • "재주문" 오표시 → 주문 취소 이력 때문
개선: 챗봇 FAQ로 에러 문구만 받아 원인+해결 즉답. 특히 학원명 특수문자/쉼표는 제품 입력검증으로 발생 자체 차단
4
예원님 응답의 상당수가 "되묻기"다
전체에 분포
  • "학원 이름 알 수 있을까요?", "어떤 단계에서 떴나요?", "클레모에서 하신 거죠?" — 정보 부족으로 인한 왕복
개선: 챗봇이 처음부터 필수 필드(학원명·사업자·카드번호·에러문구·발생화면) 강제 수집 → 되묻기 라운드 통째로 소멸
03

우리가 봐야 할 것 / 개선 방향

봐야 할 것개선 방향
온보딩 계정 생성이 최다·완전 정형셀프 온보딩 or 챗봇 잡 — 1순위. DB 분석의 잔액이전 취소보다 상위
"안 보여요" = 데이터 이원화·중복학원챗봇이 카드/학원 조회해 결정적 진단 즉답
에러 5종으로 수렴챗봇 FAQ + 학원명 입력검증으로 발생 자체 차단
예원님 되묻기 다발챗봇 필수 필드 강제 수집 (인테이크 표준화)
진선님 = 사람 라우터챗봇이 상황정리·1차진단·정형요청 생성까지 대행
핵심 한 줄 — DB 분석이 "예원님 손을 줄이는 법"이었다면, 전체 분석은 "진선님이라는 중계 단계 자체를 없애는 법"입니다. 챗봇의 진짜 목표는 단순 자동화가 아니라 고객 → (챗봇 1차 진단·정형화) → 필요한 것만 개발 큐로 흐름을 바꾸는 것입니다.
PART 2

DB 관련 작업 내역 분석 — 예원님 직접 작업

전체 중 예원님이 운영 DB를 직접 수정·삭제·반영·권한변경해야 했던 44 스레드만 별도 분석

00

개요

진선님이 예원님께 보낸 요청 중, 단순 문의가 아니라 실제로 예원님이 운영 DB를 직접 건드려야 했던 대화만 추출했습니다. "어떤 요청이 반복되는가 → 챗봇/어드민으로 무엇을 없앨 수 있는가"를 판단하기 위한 분석입니다.

분석 대상: 두 분이 함께 참여 + 진선님 요청에 DB 작업 신호 + 예원님이 실제 처리·조작한 스레드 = 총 44건

02

핵심 발견 — 진선님 요청은 "정형 데이터 전달"이 대부분

거의 모든 스레드가 동일한 패턴으로 진행됩니다.

진선 학원명 / 사업자번호 / 카드번호(원본·신규) / 원장명 / 전화번호 전달 + "처리 부탁드려요"
예원 DB에서 작업 후 institution_id / scholarship_card_id / card_balance_transfer_id 기록 회신

즉 판단이 거의 없고 식별자만 받으면 끝나는 작업이 대다수입니다. 예원님 회신의 상당수가 "처리 완료 + 기록 남기기"에 그칩니다. 이 정형성이 챗봇·어드민 자동화의 핵심 기회입니다.

03

챗봇/어드민 설계 우선순위 (소통 제거 효과 順)

각 항목을 클릭하면 접고 펼 수 있습니다. 흐린 항목은 챗봇이 아닌 기능·스케줄러 개선 영역입니다.

1
잔액이전 신청 취소
챗봇 완전 자동화
7건
  • 패턴 100% 동일: 원장이 새 카드가 아닌 사용중 카드로 잔액이전 신청 → ICB 보류 → 취소 요청 → 예원님이 card_balance_transfer 데이터 삭제 + 카드 상태 원복
  • 예원님 회신이 사실상 식별자 기록뿐 → 셀프 취소 버튼 또는 챗봇 트리거 잡(job)으로 대체하면 이 소통은 0이 됨
근본 예방: 제품에서 사용중 카드로는 잔액이전 신청 자체를 차단하면 이 유형 대부분이 발생조차 안 함 — 챗봇보다 UX 가드가 우선
2
오생성 기관/계정/주문 삭제
어드민 기능 신설
9건
  • 예원님이 직접 언급: "기관 삭제 기능이 따로 없어서 데이터 조작으로 진행"
  • 근본 원인: 실무자/원장이 기관·계정을 중복·오생성
  • 필요: 어드민 기관/계정/주문 cascade 삭제 기능 (institution · membership · order · payment · 카카오 연결까지) + 토스 환불 연동
3

5건

  • 필요: 어드민 셀프 정보 수정 + 소셜 연결 해제 + 임시 비밀번호 발급

6건

  • 해결책은 이미 예원님이 약속한 어드민 수동 반영 버튼 + 스케줄 시간 확장(19시·21시). 챗봇이 아니라 기능 개선 영역

수동 매핑 (ICB 계정 ↔ 학원)
기능 안정화 우선
6건

  • 챗봇은 절차 안내까지만 가능, 매핑 실행은 안정화된 어드민 기능 필요
6
마이그레이션 / 기관 생성

  • 챗봇은 구조화 인테이크 폼(학원/사업자/원장/전화/시나리오)으로 접수만 표준화 → 정보 누락으로 인한 왕복 제거

3건

04

챗봇 설계에 바로 쓸 결론

  • 즉시 자동화 (판단 불필요)
    잔액이전 신청 취소 — 식별자만 입력받아 잡 실행. 단일 항목으로 반복 소통 최다 제거.
  • 어드민 기능으로 흡수
    기관/계정/주문 삭제, 원장 정보 변경, 잔액이전 수동 반영 — "기능이 없어서 DB 직접 조작"이 9건+. 만들면 진선↔예원 소통의 절반 가까이 사라짐 (챗봇 아님).
  • 챗봇 = 인테이크 표준화
    마이그레이션·매핑·삭제 요청 시 필수 식별자(학원명/사업자번호/카드번호/원장명/전화)를 빠짐없이 받아 큐에 적재 → 예원님은 실행만. 정보 누락 왕복 소멸.
  • 근본 예방 (제품)
    ① 사용중 카드 잔액이전 신청 차단 ② 기관 중복 생성 방지 — 이 둘만 막아도 1·2순위 발생량이 구조적으로 감소.
